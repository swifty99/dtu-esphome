# ESPHome Hoymiles DTU — Implementation Plan

External component for Hoymiles HMS/HMT inverters (CMT2300A radio).
HM-series (NRF24) is scoped but deferred. Source code is ported "as-is"
from OpenDTU (primary) and AhoyDTU (reference); upstream is treated as
frozen.

The plan is organized as **phases** (sequential delivery milestones) plus
**cross-cutting workstreams** (CI, docs, testing, UX) that are set up
early and grow with each phase.

---

## Guiding principles

- **Don't fork the world.** Vendor only the protocol + radio code into
  `components/hoymiles_dtu/vendor/`. Keep original headers, attribution,
  and a `VENDOR_NOTES.md` mapping every file to its upstream origin and
  commit hash. This makes future cherry-picks possible without a full
  re-port.
- **Idiomatic ESPHome over upstream idioms.** Replace Arduino `Serial`,
  raw `SPI.h`, FreeRTOS tasks, `millis()` polling loops with
  `ESP_LOGx`, `spi::SPIDevice`, `Component::loop()` and
  `Scheduler::set_interval`. Anything that fights ESPHome's lifecycle
  gets rewritten.
- **One Hub, many entities.** A single `hoymiles_dtu` Hub owns the
  radio. Inverters are children. Sensors / numbers / switches reference
  inverter IDs. Mirrors how `modbus_controller`, `mqtt`, `pn532` do it.
- **Fail loud, fail early.** `setup()` calls `mark_failed()` on chip-ID
  mismatch. `dump_config()` is the diagnostic source of truth.
- **Boring releases.** SemVer, signed tags, generated changelog,
  pinned ESPHome version compatibility matrix.

## Phase 0 — Repo bootstrap

**Goal:** A pushable, lintable, compilable empty component that nobody
can break. No Hoymiles logic yet.

**Tasks**

1. Create repo, GPL-3.0 license, populate `NOTICE` with OpenDTU and
   Ahoy attributions.
2. Add `components/hoymiles_dtu/` with a no-op `Component` subclass
   that just logs `"Hello from Hoymiles DTU"` in `setup()` and
   `dump_config()`.
3. Define the *minimum* hub config schema in `__init__.py`: just `id`
   and a logger tag. Schema validation must reject unknown keys.
4. Write `tests/components/hoymiles_dtu/test.minimal.yaml` —
   compiles against ESP32 with the empty hub.
5. Tooling:
   - `.clang-format` copied from ESPHome upstream.
   - `pyproject.toml` with `ruff` (lint + format) configured to match
     ESPHome's style.
   - `.pre-commit-config.yaml` running `ruff`, `clang-format`,
     `yamllint`, `end-of-file-fixer`, `trailing-whitespace`.
   - `.editorconfig` matching ESPHome conventions.
6. Initial CI workflow `ci.yaml`:
   - **lint job**: pre-commit on changed files.
   - **compile job**: matrix [esp32, esp32s3] × [arduino, esp-idf],
     runs `esphome compile tests/components/hoymiles_dtu/test.*.yaml`
     in the `ghcr.io/esphome/esphome` container, pinned by digest.
   - **pytest job**: runs unit tests under `tests/unit/`.
7. Branch protection on `main`: require CI green, linear history,
   no force push.
8. Initial `README.md` with the unambiguous "this is alpha, no
   working radio yet" warning.

**Acceptance**

- `pre-commit run --all-files` clean.
- All matrix builds green.
- A user can already add the component via `external_components:
  github://...` and compile (no-op).

## Phase 1 — Hardware bring-up: CMT2300A detection

**Goal:** Prove the SPI plumbing works on real hardware. No Hoymiles
protocol yet — just talk to the chip.

**Tasks**

1. Extend `__init__.py`: hub now requires `cs_pin`, `gpio2_pin`,
   `gpio3_pin`, `sdn_pin`, optional `region` (EU / US).
2. Inherit `spi::SPIDevice<spi::BIT_ORDER_MSB_FIRST,
   spi::CLOCK_POLARITY_LOW, spi::CLOCK_PHASE_LEADING,
   spi::DATA_RATE_4MHZ>`.
3. Port `radio_cmt2300a` initialization sequence from OpenDTU's
   `HoymilesRadio_CMT.cpp` — *only* the parts up to and including
   reading the chip's part number register. Everything else (FIFO,
   freq hopping) is stubbed.
4. `setup()`:
   - Toggle SDN, init the chip, read part number.
   - On mismatch → `ESP_LOGE` + `mark_failed()`.
5. `dump_config()` reports: chip detected y/n, part number, region,
   pin assignments.
6. New test YAML `test.cmt2300a-bringup.yaml`.
7. **First on-hardware test point** *(deferred — runs once any tester
   is available, blocks Phase 2 merge but not Phase 1 merge).*

**Acceptance**

- CI: compiles on all matrix targets.
- Manual on-hardware (when available): `dump_config()` reports
  chip detected, no SPI errors in logs at boot.

## Phase 2 — Read-only sensors, single inverter

**Goal:** A single HMS-XXX inverter publishes live AC power, voltage,
frequency, temperature, today/total yield, and per-channel DC values.

**Tasks**

1. Port from OpenDTU `src/Hoymiles/`:
   - Frequency hopping table for the configured region.
   - Packet framing, CRC8 / CRC16, fragmentation handling.
   - Command class for `RealTimeRunData_Debug` (the standard
     "give me sensor values" request).
   - Parsers for HMS-300 / -500 / -700 / -800 / -1000 / -1600 /
     -2000 (split across files in `models/`).
2. Model definition is data-driven: a `struct` lists channel count,
   field offsets, scaling factors. Adding a new model = adding one
   struct, no parser changes.
3. `inverter.{h,cpp}`:
   - State machine: IDLE → REQUEST_SENT → AWAITING_FRAGMENTS →
     PARSED → IDLE. Timeout in each state.
   - Owns its sensor pointers; publishes via `sensor->publish_state()`.
4. Hub `loop()` schedules inverter polling round-robin; default
   `poll_interval: 15s` per inverter.
5. `sensor.py` config schema:

   ```yaml
   sensor:
     - platform: hoymiles_dtu
       inverter_id: inv1
       ac_power:    { name: "AC Power" }
       ac_voltage:  { name: "AC Voltage" }
       ac_frequency: { name: "AC Frequency" }
       temperature: { name: "Inverter Temp" }
       yield_today: { name: "Yield Today" }
       yield_total: { name: "Yield Total" }
       channel:
         - number: 1
           dc_power:   { name: "Panel 1 Power" }
           dc_voltage: { name: "Panel 1 Voltage" }
           dc_current: { name: "Panel 1 Current" }
           yield_today: { name: "Panel 1 Yield Today" }
   ```

   Auto-applies sensible `device_class`, `state_class`, `unit_of_measurement`
   for each field — user shouldn't need to specify them.
6. `text_sensor.py`: `last_seen`, `status` (`online` / `offline` /
   `producing`).
7. **First serious on-hardware tests:**
   - Boot, observe live values for ≥30 minutes.
   - Compare against AhoyDTU running on a second ESP next to ours,
     same inverter — values must match within radio jitter.
   - Force inverter offline (cover panels) → `status` flips to
     `offline` after timeout.
   - Reboot loop test: 100 reboots, no crashes, no SPI deadlock.

**Acceptance**

- All sensors populate within 60s of boot under normal radio
  conditions.
- Values match Ahoy reference within ±2%.
- No memory leaks over 24h soak (free heap stable in logs).
- New inverter model addable with <50 lines in `models/`.

## Phase 3 — Power limit (`number` entity)

**Goal:** User can set inverter active-power limit in % from Home
Assistant.

**Tasks**

1. Port `ActivePowerControl` command from OpenDTU.
2. `number.py` platform with `power_limit` field, range 2–100%, step
   1%, unit `%`.
3. Persist last set value to flash (use ESPHome's preferences API
   so it survives reboots).
4. On `control()`: send command, *don't* publish optimistic state —
   wait for the inverter's ack frame, then publish. On ack timeout,
   retry up to 3× then log warning and revert displayed value.
5. Action `hoymiles_dtu.set_limit` for use in automations
   (zero-export setups).
6. Document caveat clearly: limit is *not* zero on power loss; set
   it explicitly if needed.

**On-hardware tests**

- Set limit to 50%, observe AC power capping within ~10s.
- Set limit at boot via `on_boot` automation, verify it sticks.
- Network outage during command → no zombie state, retry resolves.

**Acceptance**

- HA shows the slider, drag → real cap.
- 1000 limit changes over 24h, no missed acks beyond 1%.

## Phase 4 — On/off switch + restart button

**Goal:** Turn the inverter on/off remotely. Reboot it.

**Tasks**

1. Port `PowerControl` and `RestartInverter` commands.
2. `switch.py` platform with `enable` field. Optimistic = no.
3. `button.py` platform with `restart` field.
4. Status text-sensor reflects on/off accurately.

**On-hardware tests**

- Toggle 50× on/off, no stuck state.
- Restart button: inverter visibly reboots (status → offline →
  online cycle).

**Acceptance**

- Same as Phase 3 — observed real-world behavior matches HA state.

## Phase 5 — Multi-inverter

**Goal:** One ESP32, one CMT2300A, multiple HMS inverters.

**Tasks**

1. Hub `inverters` is already a list — verify scheduler fairness
   under N inverters.
2. Per-inverter poll budgeting: `total_poll_budget = poll_interval`,
   each inverter gets `poll_interval / N` worth of radio time per
   cycle.
3. Tune frequency-hop timing so back-to-back inverter polls don't
   stomp on each other.
4. Stress test with 2 real inverters if available; otherwise simulate
   the second via a recorded packet replay (see Testing section).

**Acceptance**

- 2 inverters polled every 15s without dropouts > 5%.
- `dump_config()` clearly enumerates all inverters and their state.

## Phase 6 — HM-series (NRF24L01+) *(deferred, separate milestone)*

**Goal:** Same feature set as Phase 2–4 for HM-300/600/800/1000/1200/
1500 via NRF24.

**Tasks**

1. Add `radio/radio_nrf24.{h,cpp}`, port from OpenDTU.
2. Hub gains a `radio: nrf24 | cmt2300a | both` selector. `both`
   requires ESP32 with two SPI buses (HSPI + VSPI).
3. Per-inverter `radio: nrf24 | cmt2300a` so the hub knows which
   modem to route a given serial through.
4. Reuse `inverter.{h,cpp}` and `protocol/` — only the radio layer
   differs.

Treat as a separate, self-contained milestone. Don't block 1.0 on it.

## Cross-cutting: CI/CD

**`ci.yaml` (PR + push to main)**
- Triggered on all PRs and pushes to `main`.
- Jobs:
  - **`pre-commit`**: lints all changed files. Cancels other jobs
    if it fails.
  - **`pytest`**: runs `tests/unit/` on Python 3.11 + 3.13.
  - **`esphome-compile`**: matrix over (esp32 / esp32s3 / esp32c3) ×
    (arduino / esp-idf) × (test files in `tests/components/`).
    Runs in `ghcr.io/esphome/esphome:<pinned>` container. Cache
    `.platformio` between runs keyed on platformio.ini hash.
  - **`memory-check`** *(optional, advisory)*: log flash + RAM delta
    vs. `main`. Posts comment on PR.
- ESPHome version pinned in one place (`.github/esphome-version.txt`),
  consumed by all jobs and `docs/getting-started.md`.

**`release.yaml` (tag-triggered)**
- On tag `v*.*.*`:
  - Generate changelog with `release-please` or similar.
  - Create GitHub Release with assets (none needed — repo *is* the
    artifact).
  - Push doc rebuild.

**`docs.yaml` (push to main, paths: docs/**)**
- `mkdocs build && mkdocs gh-deploy`. Use Material theme — clean,
  light, default-readable.

**`stale.yaml`**
- Close issues with no activity in 90d after a 14d warning.
  Skip `pinned` and `bug` labels.

**Dependabot**
- Watches GitHub Actions versions and pre-commit hook versions.
  Weekly schedule. Auto-merge on green CI for patch bumps only.

## Cross-cutting: Testing

Three layers, mapped to where bugs actually live.

**Layer 1 — Unit tests (pytest, fast, run on every commit)**
- `test_crc.py`: known good vectors for CRC8/CRC16.
- `test_parser.py`: feed captured packet bytes (hex dumps from real
  inverters, kept as fixtures), assert decoded sensor values.
- `test_schema.py`: import the component, run `CONFIG_SCHEMA` against
  good and bad YAML — bad must raise.

**Layer 2 — Compile tests (CI matrix, ~minutes)**
- Each YAML in `tests/components/hoymiles_dtu/` must compile clean
  on every supported (board × framework) combo.
- Add a YAML for *every public config shape* in the docs. If the
  docs show it, CI proves it builds.

**Layer 3 — Hardware-in-the-loop (HIL, manual, gated to release)**
- A short, scripted HIL checklist in
  `tests/hil/RELEASE_CHECKLIST.md` that the maintainer ticks off
  before each tagged release on the real ESP32 + HMS rig:
  - Cold boot, all sensors populate within 60s.
  - 12-hour soak, free heap remains stable (logged via API).
  - Limit ramp 100% → 20% → 100%, observe AC power tracks.
  - On/off cycle.
  - Reboot resilience: power-cycle ESP mid-poll, recovers cleanly.
- Optionally automate with a tiny `pytest-aioesphomeapi` harness
  that connects to the device's native API and asserts state. This
  is a stretch goal — the manual checklist is sufficient for 1.0.

**Packet capture corpus**
- Fixtures dir: `tests/fixtures/packets/<inverter_model>/<scenario>/`.
- Whenever an issue gets solved with a captured packet, it joins
  the corpus. Tests replay it. Regressions get caught.

## Cross-cutting: Documentation

**Site:** MkDocs Material, deployed to GitHub Pages on push to
`main`. Default light theme, no dark by default — matches ESPHome's
own docs style.

**Pages mirror the code's surface, not the code's structure:**

- **Getting Started**: 3 sections — wiring, minimal YAML, first
  flash. Anyone with an ESP32, a CMT2300A, and an HMS inverter
  must reach "values in HA" inside 30 minutes.
- **Hardware**: wiring diagrams (Fritzing or hand-drawn — clarity
  beats prettiness), recommended boards, antenna notes, dual-SPI
  pin maps for Phase 6.
- **Configuration**: one page per platform (hub / sensor / number /
  switch / button). Each option: type, default, range, example,
  caveat.
- **Examples**: full working YAMLs for common scenarios. These
  *are* the same files in `examples/`, included via mkdocs
  snippets — single source of truth.
- **Supported inverters**: table, source = `models/` directory,
  generated by a small script in CI to prevent doc drift.
- **Troubleshooting**: symptom → cause → fix matrix. "Sensors stuck
  at zero", "values jitter wildly", "limit doesn't apply",
  "compile fails on ESP-IDF".
- **Attribution**: explicit list of files derived from OpenDTU and
  Ahoy with upstream commit hashes.

**Inline:**
- Every public class in C++: doxygen-style header.
- Every Python config option: docstring with type, range, default,
  HA-side effect.

**Code-level docs are not aspirational.** CI fails if
`models/<new_model>.h` lacks the required header block. Enforced
with a tiny `scripts/check_model_headers.py`.

## Cross-cutting: Usability

This is the part that determines whether anyone other than the
author uses the component.

**YAML ergonomics**
- Sensible defaults everywhere. The minimal "I have an HMS-800-2T
  at serial 114180000000" config should be ~15 lines including
  `spi:` block.
- Per-sensor naming follows ESPHome convention: omit `name:` and
  the entity inherits the hub-level prefix.
- Friendly errors: schema validation messages name the offending
  key + offer the closest match. Use `cv.one_of` with `lower=True`
  for enum-like fields.
- `id:` for everything is auto-generated when omitted. User shouldn't
  hand-name an ID just to satisfy a schema.

**HA-side defaults**
- Pre-set `device_class`, `state_class`, `unit_of_measurement`,
  `accuracy_decimals`, `icon` for every sensor. User just adds a
  name; the entity card looks correct in HA on first sight.
- `entity_category: diagnostic` for chip stats, `last_seen`,
  `status`. Production values stay in the main entity list.
- Group all entities under one HA device automatically (ESPHome
  does this — just ensure naming is consistent).

**Diagnostics**
- `dump_config()` is verbose: inverter list, current state per
  inverter, last seen, signal indicator if available, current
  limit, region, radio chip ID. The first thing a user pastes into
  an issue is the boot log — it should already contain everything
  the maintainer needs.
- Error log lines are unique strings (no two error sites share a
  message), so users can grep, and the maintainer can locate
  source instantly.

**Onboarding**
- A "1-click" config: `examples/basic-hms-800.yaml` is a complete
  working file with placeholders (`!secret wifi_ssid`,
  `<INVERTER_SERIAL>`). Comments explain every non-trivial line.
- A second example, `zero-export-with-shelly.yaml`, shows the
  *real* reason most people install this: tracking household
  consumption from a Shelly 3EM and feeding it into the limit
  number.
- `examples/multi-inverter.yaml` for the Phase 5 case.

**Issue templates**
- `bug.yaml`: forces version, ESPHome version, board, framework,
  inverter model, full boot log. No more "doesn't work" issues.
- `inverter-support.yaml`: model number, capture file (link to
  AhoyDTU instructions for capturing one), known channel count.
  This is how Phase-2's `models/` grows over time.

**Versioning & deprecation**
- SemVer. Schema changes that break existing YAML = major bump.
- Deprecation cycle: warn for one minor version, remove on next
  major. Warnings come from `cv.invalid` with a clear migration
  hint.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| CMT2300A timing differs subtly under ESPHome's cooperative loop vs. OpenDTU's tasks | Keep radio-critical sections short, never `delay()` > 1ms in `loop()`, measure with logic analyzer if values flap |
| Solo maintainer fatigue | Aggressive automation (CI, release-please, dependabot, stale bot) + low-friction issue templates so most issues become 5-minute triage |
| Upstream protocol updates we miss | `VENDOR_NOTES.md` per file with upstream commit hash; a calendar reminder every 3 months to diff against upstream main |
| ESPHome breaking changes | Pin tested ESPHome version; CI matrix includes `dev` as advisory (allowed-to-fail); compatibility matrix in docs |
| Lic. accusations re: vendored code | NOTICE file, file-level headers, separate `vendor/` directory, GPL-3.0 repo license, no marketing claims of original work |

## Definition of "1.0"

- All Phase 0–5 acceptance criteria green.
- HIL release checklist completed on real hardware.
- Docs site live, every option documented, troubleshooting page
  has at least 5 entries.
- ≥3 distinct users (besides the maintainer) confirmed working
  setups via issues or discussions.
- A community member somewhere has submitted a model addition via
  the `inverter-support` template — this proves the contribution
  path works.

Until that last point is true, ship as `0.x` and treat the API as
unstable.
